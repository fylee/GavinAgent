# 021-1 — Skill Authoring Compliance (amends Spec 020)

## Goal

Update the Claude Code skill authoring system (Spec 020) to produce SKILL.md files that comply with the agentskills.io specification and align with the routing architecture decided in Spec 021.

## Dependency

**Must be implemented after Spec 021.** This spec assumes:
- The frontmatter schema from 021 (non-spec fields in `metadata`)
- The hybrid routing decision from 021 (embedding + catalog; `description` serves both)

## Background

Spec 020 defined a skill authoring system where Claude Code writes SKILL.md files following GavinAgent conventions. However, those conventions pre-date the agentskills.io specification. If 020 is implemented as written, Claude Code will author skills with non-compliant frontmatter (top-level `triggers`, `examples`, `version`), undoing the work of Spec 021.

Three specific problems in Spec 020:

1. **AGENTS.md schema is wrong** — the planned `## Skill Authoring` section lists `triggers`, `examples`, `version` as top-level YAML keys, which 021 prohibits
2. **Review criteria are wrong** — `review_skill()` evaluates skills against GavinAgent conventions, not the agentskills.io spec; a "passing" review can still produce non-compliant output
3. **Description quality is not emphasised** — the authoring prompt doesn't guide Claude Code to write descriptions that serve as routing signals; in the hybrid model, a good `description` is required for catalog routing

## Proposed Changes

### 1. Update AGENTS.md `## Skill Authoring` schema

Replace the schema section with the 021-compliant frontmatter:

```markdown
## Skill Authoring

### SKILL.md frontmatter schema

Required fields:
- `name`: lowercase, hyphens only, matches directory name, max 64 chars
- `description`: 1–1024 chars; must describe **what** the skill does AND **when** to use it;
  include specific task keywords so agents can identify relevance

Optional spec fields:
- `allowed-tools`: space-separated tool names pre-approved for this skill
- `compatibility`: environment requirements (network, packages, MCP servers)
- `license`: license name or file reference
- `metadata`: arbitrary key-value map (string values only)

GavinAgent extensions (must live inside `metadata`, not at top level):
- `metadata.triggers`: pipe-separated keywords for embedding routing (e.g. `"wip | movement | lot"`)
- `metadata.trigger_patterns`: pipe-separated regex patterns for phrase matching
- `metadata.examples`: pipe-separated example user requests
- `metadata.version`: skill version string (e.g. `"1"`)
- `metadata.approval_required`: `"true"` or `"false"`

❌ Do NOT write these as top-level keys — they are non-compliant:
  triggers: [...]      # wrong
  examples: [...]      # wrong
  version: 1           # wrong

✅ Correct:
  metadata:
    triggers: "wip | movement"
    version: "1"
```

### 2. Update authoring prompt — description quality

Add to the authoring prompt passed to Claude Code:

```
The `description` field is the most important field in the skill.
It must:
- Describe what the skill does (capabilities, outputs)
- State when to use it (trigger conditions, user intent signals)
- Include domain-specific keywords an agent would recognise
- Be 80–300 characters (not too short, not exhaustive)

Good example:
  "Analyse tabular data, compute statistics, rank and filter datasets.
   Use when the user asks to calculate, compare, aggregate, or summarise
   numerical data from a table, CSV, or query result."

Poor example:
  "Helps with data."
```

### 3. Update `review_skill()` checklist

The review function must validate against two checklists:

**Spec compliance checklist (agentskills.io):**
- [ ] Only spec-defined top-level keys present
- [ ] `name` matches parent directory name
- [ ] `name` is lowercase, no consecutive hyphens, no leading/trailing hyphen
- [ ] `description` is non-empty and ≥ 50 characters
- [ ] All GavinAgent extensions are inside `metadata`
- [ ] `metadata` values are strings (not lists or booleans)

**Routing quality checklist (GavinAgent hybrid model):**
- [ ] `description` contains both capability keywords and usage trigger keywords
- [ ] `metadata.triggers` covers the main vocabulary a user would use for this skill's domain
- [ ] No overlap between this skill's `triggers` and other installed skills (check for collision)

### 4. Add `skills-ref validate` step to review workflow

After Claude Code writes or rewrites a SKILL.md, the review workflow should run:

```bash
uv run skills-ref validate agent/workspace/skills/<name>
```

If validation fails, Claude Code must fix the reported issues before marking the review complete.

## Out of Scope

- Changing any existing SKILL.md content (that is Spec 021's job)
- Changing Mode A/B subprocess vs MCP architecture (Spec 020's implementation detail)
- Implementing the catalog injection into the system prompt (that is Spec 022's job)

## Acceptance Criteria

- [ ] AGENTS.md `## Skill Authoring` section uses 021-compliant frontmatter schema
- [ ] Top-level `triggers`, `examples`, `version`, `approval_required` are absent from the schema documentation
- [ ] Authoring prompt instructs Claude Code to write descriptions that cover both "what" and "when"
- [ ] `review_skill()` runs both the spec compliance checklist and routing quality checklist
- [ ] `skills-ref validate` is invoked as part of the review workflow
- [ ] A skill authored by Claude Code after this spec is implemented passes `skills-ref validate` without errors
